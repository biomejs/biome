修复lint规则、格式化布局等可能会阻止你的脚本通过。由于这些更改的性质，
我们**强烈建议**在你的`package.json`中保存_确切_的版本，而不是使用范围操作符。

这种方法将确保你的脚本不会意外失败。

## 语义版本控制

Biome遵循[语义版本控制](https://semver.org/)。由于Biome作为一个工具链的性质，可能不清楚哪些更改被认为是主要的、次要的或补丁。这就是为什么Biome使用以下版本控制指南：

### 补丁发布

* 修复一个lint规则，该规则对有效代码提出lint错误（假阳性）
* 修复错误的代码建议
* 修复格式化语法，导致代码无效或改变程序的语义。
* 对文档的改进
* 不改变Biome功能的内部更改：
  * 重构
  * 性能改进
  * 增加或改变测试覆盖率
* 改进诊断的措辞或修复诊断的呈现。
* 发布失败后的重新发布
* 改变已建立语法的格式化。

### 次要发布

* 添加一个新的规则或将现有的lint规则提升到默认不推荐的稳定组。
* 为最近引入的语言特性添加lint和格式化支持，即使这导致更多的lint错误报告。
* 删除推荐的规则
* 废弃现有的规则
* 添加新的配置可选配置选项，不改变格式化或报告更多的lint错误。
* 添加一个新的推荐lint规则或将现有的lint规则从nursery组提升到稳定组的推荐lint规则。
* 删除非*nursery*规则或将规则降级到*nursery*组。

### 主要发布
* 更改配置，导致不同的格式化或更多的lint错误报告（添加/删除选项，更改默认值）
* 更改Biome的公共API
* 需要一些聚光灯的新特性或工具的推广

## Visual Studio Code扩展

Visual Studio Code [不支持预发布标签](https://code.visualstudio.com/api/working-with-extensions/publishing-extension#prerelease-extensions)的扩展。这就是为什么Biome使用以下版本模式来区分稳定和预览：
* 稳定版本使用偶数版本号：10、12、14、16、...
* 预览使用奇数版本号：11、13、15、17、...
